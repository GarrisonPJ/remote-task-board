Extract the Authorization Policy module

Worth exploring in-process

Files

services/task-service.ts · services/member-service.ts

Before — Inlined checks

task-service.ts

if (!session?.user) throw "Unauthorized"

member-service.ts

if (role !== "ADMIN") throw "Only admins..."

member-service.ts

if (role !== "ADMIN") throw "Only admins..."

Same check duplicated. No task-level permissions.

After — Declarative policy

lib/authorization.ts

authorize("task:create", ctx)
authorize("task:delete", ctx)
authorize("member:invite", ctx)
authorize("member:remove", ctx)

// Policy Map
task:create  → [ADMIN, MEMBER]
task:delete  → [ADMIN] or isCreator
member:invite → [ADMIN]
member:remove → [ADMIN]

One interface. One place to test. One place to change.

Problem: Role checking inlined across services; no seam between "who can" and "the doing"; no task-level permissions.

Solution: Declarative policy map behind a single authorize() interface.

  • ✦ Locality: all permission rules in one module
  • ✦ Leverage: one interface, every protected action
  • ✦ Pure function — testable with zero mocks
  • ✦ Delete 3 inline permission checks
  • ✦ Enables Viewer Role without touching services

Problem: No error module exists; errors silently logged; crashes destroy the page.

Solution: Three-layer defence: structured responses → Error Boundary → Suspense skeleton.

  • ✦ Leverage: one Error Boundary catches all component crashes
  • ✦ Locality: error presentation in one place, not per-component
  • ✦ Members see feedback instead of console.error
  • ✦ Failed optimistic updates roll back visually

Top Recommendation

Deepen the Task mutation pipeline — tackle first.

It has the highest leverage: every future mutation (Activity Log writes, new Task operations) passes through it. It unblocks the Authorization Policy module (which plugs into the pipeline) and the Activity Log (which emits from it). And it's the lowest-risk refactor — the external interface doesn't change, so all existing tests pass untouched.

Pair with the Sync Event extraction (candidate #2) for the highest compound impact in a single sprint.
